Implementation Approach

3 strategies for delivering the Option-B glass upgrade. Trade-offs shown below.

Option A

CSS-Only Sweep

Apply glass classes globally via index.css and Tailwind. Minimal per-file edits.

  • Extend .glass-card / .glass-panel with B-level depth
  • Apply to TopNav, BottomNav, dialogs via global selectors
  • No per-page JSX changes
Risk: Low · Fastest

⚠️ Can't target individual pages precisely. Dialogs & table cards need JSX edits to actually use glass classes.

Option B

Layered File-by-File ★ Recommended

Upgrade each surface in its own file. Design tokens in CSS, glass applied precisely where needed.

  • Enhanced design tokens in index.css
  • TopNav + BottomNav → glass treatment
  • Tables page → glass cards + skeleton
  • Dashboard → glass stat cards
  • Settings / Admin → glass panels + sections
  • Dialogs → glass DialogContent override
  • POS top bar + cart panel → glass only (no logic touch)
Risk: Medium · 1–2 days

Best balance of coverage, safety, and visual impact. Every file edit is UI-layer only.

Option C

Component Extraction

Extract reusable <GlassCard>, <GlassPanel>, <StatusBadge> components. Refactor all pages to use them.

  • New shared components under client/src/components/ui/
  • All pages refactored to import & use them
  • Maximum future consistency
Risk: High · 3–5 days

Over-engineered for a UI polish task. Extraction adds unnecessary abstraction and risk of merge conflicts.
